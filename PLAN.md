## Этап 0. Подготовка проекта

1. Обновить структуру каталогов:
   ```
   ChatList/
   ├── main.py          # точка входа и GUI
   ├── db.py            # работа с SQLite
   ├── models.py        # бизнес-логика моделей и промтов
   ├── network.py       # HTTP-запросы к API нейросетей
   ├── requirements.txt
   ├── .env.example     # шаблон переменных окружения для API-ключей
   └── DATABASE.md
   ```
2. Обновить `requirements.txt`: PyQt6, httpx (или requests), python-dotenv.
3. Создать `.env.example` с примерами имён переменных (`OPENAI_API_KEY`, `DEEPSEEK_API_KEY` и т.д.).
4. Добавить в `.gitignore` уже существующие правила + убедиться, что `.env` игнорируется.

## Этап 1. База данных (`db.py`)

1. Реализовать подключение к SQLite (файл `chatlist.db` в каталоге приложения).
2. Создать функцию инициализации схемы (`init_db`) — таблицы `prompts`, `models`, `results`, `settings` (см. DATABASE.md).
3. Реализовать CRUD-операции для каждой таблицы:
   - **prompts** — добавление, список, поиск, удаление;
   - **models** — добавление, список активных, редактирование, включение/выключение;
   - **results** — сохранение выбранных строк, список, поиск;
   - **settings** — чтение/запись пар ключ–значение.
4. Добавить начальные данные (seed): 2–3 модели-заглушки с `is_active=0`, чтобы пользователь мог их настроить.
5. Написать простой тест запуска `init_db` из консоли.

## Этап 2. Сетевой слой (`network.py`)

1. Реализовать базовый класс или функцию отправки промта в API по URL и идентификатору модели.
2. Загружать API-ключ из `.env` по имени переменной, указанному в записи `models.api_key_env`.
3. Реализовать адаптеры для разных типов API (минимум OpenAI-совместимый формат):
   - OpenAI / OpenAI-compatible (DeepSeek, Groq и др.);
   - единый интерфейс: `send_prompt(model_row, prompt_text) -> str`.
4. Обработать ошибки сети и API (таймаут, 401, 429) — возвращать текст ошибки вместо ответа модели.
5. Добавить параллельную отправку запросов ко всем активным моделям (потоки или `asyncio` + httpx).

## Этап 3. Бизнес-логика (`models.py`)

1. Обернуть вызовы `db.py` в функции высокого уровня:
   - `get_active_models()`;
   - `save_prompt(text, tags)`;
   - `get_prompts(search=None)`;
   - `save_selected_results(prompt_id, rows)`;
   - `get_settings()` / `set_setting(key, value)`.
2. Реализовать сценарий «отправить промт»:
   - получить активные модели;
   - вызвать `network.send_prompt` для каждой;
   - вернуть список словарей `{model_name, response, selected}` для временной таблицы.
3. Реализовать очистку временного состояния при новом запросе (управление in-memory списком).

## Этап 4. Графический интерфейс (`main.py`)

### 4.1. Главное окно

1. Заменить текущую минимальную заготовку на полноценное окно ChatList.
2. Разделить окно на зоны:
   - **верх** — ввод промта (QTextEdit) + кнопка «Отправить»;
   - **слева/выпадающий список** — выбор сохранённого промта из БД;
   - **центр** — временная таблица результатов (QTableWidget):
     - колонки: «Модель», «Ответ», «Выбрать» (QCheckBox);
   - **низ** — кнопки «Сохранить выбранные», «Новый запрос».

### 4.2. Сценарии пользователя

1. **Новый промт**:
   - пользователь вводит текст → нажимает «Отправить»;
   - показывается индикатор загрузки;
   - временная таблица заполняется ответами.
2. **Выбор сохранённого промта**:
   - выбор из ComboBox → текст подставляется в поле ввода → «Отправить».
3. **Сохранение**:
   - отмеченные чекбоксы → «Сохранить» → запись в `results`, очистка временной таблицы.
4. **Новый запрос**:
   - очистка поля ввода и временной таблицы.

### 4.3. Дополнительные окна (базовая версия)

1. Окно управления моделями: список, добавление, редактирование, переключатель `is_active`.
2. Окно истории сохранённых результатов (просмотр таблицы `results`).

## Этап 5. Связка компонентов

2. При старте приложения вызывать `init_db()`.
3. Загружать `.env` через `python-dotenv` до первого сетевого запроса.
4. Проверить полный сценарий:
